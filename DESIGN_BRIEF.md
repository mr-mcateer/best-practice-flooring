## 1. Executive Summary

Best Practice Flooring LLC is a two-person, husband-and-wife flooring company based in Independence, OR, serving the greater Corvallis region (including Independence, Monmouth, Dallas, Salem, Albany, Philomath, and surrounding Polk & Benton County communities). This brief defines the strategy, design, and implementation plan for a world-class informational website — built as if a top-2% consulting firm conducted the engagement.

**Strategic Objective:** Become the top-cited flooring installer in the Corvallis/Mid-Willamette Valley region across both traditional search engines and AI answer engines (Google AI Overviews, Bing Copilot, ChatGPT Search, Perplexity).

### 3.1 Mid-Willamette Valley / Corvallis Flooring Market (2026)

- Independence / Corvallis area: affordable mid-valley market, strong OSU community
- Key trends: sustainability, warm earthy hues (honeyed oak, caramel walnut, rich chestnut)

- College town (Corvallis/OSU) + rural communities = diverse customer base

### 3.2 AI Search Landscape (AEO — Answer Engine Optimization)

AEO is the practice of structuring content so AI platforms deliver it as the direct answer. For local service businesses:
- Create structured pages with NAP (name, address, phone), services, hours, FAQs
- E-E-A-T signals critical: Experience, Expertise, Authoritativeness, Trustworthiness
- Content freshness weighted ~40% by Perplexity
- Authority signals ~15% of ranking influence
- FAQ schema and how-to schema improve AI citation rates

### 3.3 Core Web Vitals 2026 Thresholds

| Metric | Good | Needs Improvement | Poor |
|---|---|---|---|
| LCP (Largest Contentful Paint) | < 2.5s | 2.5s - 4.0s | > 4.0s |
| INP (Interaction to Next Paint) | < 200ms | 200ms - 500ms | > 500ms |
| CLS (Cumulative Layout Shift) | < 0.1 | 0.1 - 0.25 | > 0.25 |

43% of sites fail the INP threshold in 2026. A static site will crush this metric.

### 3.4 GitHub Pages Suitability

- Informational business websites: **Allowed**
- E-commerce / transactions: **Not allowed** (we won't need this)
- Free HTTPS, custom domain support
- Bandwidth: 100GB/month soft limit (more than sufficient)
- Build: GitHub Actions auto-deploy on push

## 4. Technical Architecture

### 4.1 Stack Decision

| Component | Choice | Rationale |
|---|---|---|
| **Static Site Generator** | Eleventy (11ty) | Lower learning curve, Markdown content, JS ecosystem, non-dev maintainable |
| **CSS Framework** | Tailwind CSS v4 | CSS-first config, native dark mode, design tokens via CSS variables |
| **Hosting** | GitHub Pages | Free, HTTPS, custom domain, GitHub Actions CI/CD |
| **Images** | Self-hosted from Unsplash/Pexels | Royalty-free, locally hosted for performance + no third-party deps |
| **Fonts** | System font stack + 1 display font (self-hosted) | Zero FOUT, minimal LCP impact |
| **Analytics** | Plausible or none (privacy-first) | GDPR/CCPA compliant, no cookie banner needed |
| **Forms** | Formspree or Netlify Forms (free tier) | No backend required |

## 5. Design System Specification

### 5.1 Color Blind Safe Palette

All colors tested against protanopia, deuteranopia, and tritanopia using APCA contrast standards.

#### Light Mode (Default)

```css
:root {
  /* Surface colors */
  --surface-0: #FFFFFF;         /* Page background */
  --surface-1: #F8F6F3;         /* Card background (warm white) */
  --surface-2: #EDE9E3;         /* Elevated surfaces */

  /* Text colors */
  --text-primary: #1A1714;       /* Headings — near-black warm */
  --text-secondary: #4A4540;     /* Body text */
  --text-muted: #7A756E;         /* Captions, metadata */

  /* Brand colors (color-blind safe) */
  --brand-primary: #2B6CB0;      /* Blue — safe for all CVD types */
  --brand-primary-hover: #1E4E8C;
  --brand-accent: #C05621;       /* Warm amber/orange — distinguishable from blue */
  --brand-accent-hover: #9C4216;

  /* Semantic colors */
  --success: #276749;            /* Dark green (not relying on hue alone) */
  --warning: #C05621;            /* Amber */
  --error: #C53030;              /* Red with sufficient contrast */

  /* Borders */
  --border-default: #D4CFC7;
  --border-strong: #A09A91;
}
```

#### Dark Mode

```css
[data-theme="dark"] {
  /* Layered darkness — NOT pure black */
  --surface-0: #121110;          /* Page background */
  --surface-1: #1E1D1B;          /* Card background */
  --surface-2: #2A2826;          /* Elevated surfaces */

  /* Text — APCA Lc 90+ for body text */
  --text-primary: #F0EDE8;       /* Headings */
  --text-secondary: #C8C3BA;     /* Body text */
  --text-muted: #8A847C;         /* Captions */

  /* Brand colors (lightened for dark bg) */
  --brand-primary: #63B3ED;      /* Lighter blue */
  --brand-primary-hover: #90CDF4;
  --brand-accent: #ED8936;       /* Lighter amber */
  --brand-accent-hover: #F6AD55;

  /* Semantic */
  --success: #68D391;
  --warning: #ED8936;
  --error: #FC8181;

  /* Borders */
  --border-default: #3D3A36;
  --border-strong: #5A5650;
}
```

**Color-Blind Safety Rules:**
1. Never use color alone to convey information (always pair with icons, text, or patterns)
2. Blue + Orange primary palette is distinguishable across all CVD types
3. All text meets WCAG 2.2 AA minimum (4.5:1 normal, 3:1 large)
4. Dark mode meets APCA Lc 75+ for body, Lc 90 preferred
5. Interactive elements have visible focus indicators (not color-dependent)

### 5.2 Typography

| **Headings** | Logical H1→H6 hierarchy, no skipped levels |
| **Images** | Meaningful `alt` text; decorative images use `alt=""` |
| **Forms** | Visible labels (not just placeholder), error messages linked to fields |
| **Motion** | Respect `prefers-reduced-motion` — disable animations |
| **Keyboard** | Full site navigable via Tab/Shift+Tab/Enter/Escape |
| **Screen Reader** | Tested with VoiceOver (macOS/iOS) and NVDA (Windows) |
| **Theme Toggle** | `role="switch"`, `aria-checked`, announces state change |
| **Phone Links** | `<a href="tel:+15038841711">` with visible text |

### 9.3 Dark Mode Implementation

```
User visits site
      │
      ▼
Check OS preference (prefers-color-scheme)
      │
      ├── OS = dark → Apply dark theme
      ├── OS = light → Apply light theme
      └── No preference → Light (default)
      │
      ▼
Check localStorage for override
      │
      ├── Override exists → Use stored preference
      └── No override → Use OS preference
      │
      ▼
User clicks theme toggle
      │
      ├── Save to localStorage
      ├── Update data-theme attribute on <html>
      └── Announce change to screen readers
```

**Technical:** Use `data-theme` attribute on `<html>` with CSS custom properties. Tailwind v4 `@custom-variant` for dark mode. Inline script in `<head>` to prevent flash (FOUC).

## 13. Guardrails & Constraints

### 13.1 Technical Guardrails

| Rule | Rationale |
|---|---|
| **No JavaScript frameworks (React, Vue, etc.)** | Static site; JS adds bundle size and CWV risk |
| **Maximum 15KB JavaScript total** | Theme toggle + mobile nav + form validation only |
| **All images must be WebP with JPEG fallback** | Performance + broad compatibility |
| **No third-party CDNs for critical assets** | Self-host everything for reliability + privacy |
| **No Google Fonts CDN** | Use system fonts or self-host to avoid render-blocking |
| **No cookies** | Eliminates need for cookie banner |
| **No tracking pixels** | Privacy-first; use server-side analytics if needed |
| **GitHub Pages only** | Client requirement; no server-side capabilities |
| **All forms via external service** | Formspree/Netlify Forms free tier |

### 13.2 Content Guardrails

| Rule | Rationale |
|---|---|
| **No pricing on site** | Flooring prices vary by material, area, condition |
| **No "cheap" or "discount" language** | Position as premium/quality, not lowest price |
| **Every page must have a CTA** | Conversion-focused design |
| **Phone numbers on every page** | Reduce friction to contact |
| **No stock photos of people** | Use real photos of Ake/Sarah when available |
| **Placeholder images must be flooring-relevant** | No generic business stock photos |
| **Service area must be specific** | List actual cities: Independence, Monmouth, Dallas, Salem, Corvallis, Albany, Philomath, etc. |
| **FAQs must answer real questions** | Written to match actual search queries |

### 13.3 Design Guardrails

| Rule | Rationale |
|---|---|
| **Maximum 2 fonts** | Performance + visual consistency |
| **Maximum 3 brand colors + neutrals** | Simplicity; color-blind safety |
| **No animations > 300ms** | Reduced motion support + performance |
| **No parallax scrolling** | Accessibility + mobile performance |
| **No carousels/sliders** | UX research shows low engagement |
| **No modal popups** | Especially not on mobile |
| **Minimum 16px body text** | Readability on all devices |
| **Maximum line length 75 characters** | Readability standard |

### 13.4 Process Guardrails

| Rule | Rationale |
|---|---|
| **All code must pass linting** | Consistent code quality |
| **All changes via pull request** | Code review before merge |
| **Deploy only from `main` branch** | Single source of truth |
| **Accessibility tested before merge** | Shift-left a11y |
| **No direct commits to `main`** | Force code review |

## 14. Ambiguity Resolution Protocol

When agents encounter unclear requirements or conflicting information, follow this decision tree:

### 14.1 Decision Matrix

```
┌─────────────────────────────────────────────────────────┐
│                  AMBIGUITY DETECTED                      │
└──────────────────────────┬──────────────────────────────┘
                           │
              ┌────────────┴────────────┐
              │ Is it about CONTENT?     │
              │ (copy, messaging, tone)  │
              └────────────┬────────────┘
                     YES   │   NO
              ┌────────────┘   └────────────┐
              ▼                              ▼
    ASK AKE & SARAH              ┌──────────────────────┐
    (Content must be             │ Is it a TECHNICAL     │
    owner-approved)              │ decision?             │
                                 └──────────┬───────────┘
                                      YES   │   NO
                                 ┌──────────┘   └──────────┐
                                 ▼                          ▼
                    ┌──────────────────────┐    ┌──────────────────────┐
                    │ Does it affect        │    │ Is it a DESIGN       │
                    │ ACCESSIBILITY?        │    │ decision?            │
                    └──────────┬───────────┘    └──────────┬───────────┘
                         YES   │   NO                 YES   │
                    ┌──────────┘   └────┐         ┌────────┘
                    ▼                    ▼         ▼
           CHOOSE THE MORE       CHOOSE THE    CHOOSE THE
           ACCESSIBLE OPTION     SIMPLER       SIMPLER
                                 APPROACH      APPROACH
```

### 14.2 Specific Ambiguity Scenarios

| Scenario | Resolution | Rationale |
|---|---|---|
| **Unsure about service details** | Use placeholder text with `[TODO: Confirm with Ake]` markers | Don't guess at trade expertise |
| **Two valid CSS approaches** | Choose the one with better browser support | Compatibility > cleverness |
| **Image doesn't have good alt text** | Write descriptive alt based on visible content | Never use `alt=""` for meaningful images |
| **Unsure if section is needed** | Include it but make it easy to remove | Additive is easier to undo than omission |
| **Color contrast is borderline** | Choose the higher-contrast option | Always err toward accessibility |
| **Mobile layout unclear** | Stack vertically, single column | Mobile-first = simplest layout wins |
| **Feature request seems complex** | Build the simplest version first | YAGNI — don't over-engineer |
| **Design spec conflicts with a11y** | Accessibility wins, always | Legal requirement + ethical obligation |
| **Client hasn't provided content** | Write professional placeholder, mark `[DRAFT]` | Don't block development on content |
| **Performance vs visual richness** | Performance wins for initial load | Use lazy loading for below-fold richness |

### 14.3 Escalation Protocol

1. **Self-Resolve:** If the ambiguity fits a scenario above, resolve using the prescribed approach
2. **Peer Consult:** If another agent has relevant expertise, consult their RACI role
3. **Document & Flag:** Log the decision in a `DECISIONS.md` file with rationale
4. **Escalate to Client:** If the decision materially affects the user experience or business positioning, ask Ake & Sarah

```markdown

- **Agent:** [Which agent made the call]
- **Ambiguity:** [What was unclear]
- **Options Considered:** [A, B, C]
```

## Appendix B: GitHub Pages Deployment

### deploy.yml (GitHub Actions)

```yaml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]
permissions:
  contents: read
  pages: write
  id-token: write
concurrency:
  group: "pages"
  cancel-in-progress: false
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "20"
          cache: "npm"
      - run: npm ci
      - run: npm run build
      - uses: actions/upload-pages-artifact@v3
        with:
          path: "_site"
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - id: deployment
        uses: actions/deploy-pages@v4
```
